# Homebrew Binary Release Assets Design

## Goal

Switch `numi` Homebrew installation from source builds to prebuilt GitHub release assets generated by the `numi` release workflow.

## Scope

This design covers:

- the `numi` release asset contract in `oops-rs/numi`
- the `numi` formula shape in `oops-rs/homebrew-tap`
- the tap update script logic used during releases

This design does not cover:

- crates.io publication
- Windows package managers
- Homebrew bottle generation

## Facts

- `numi` already publishes per-target archives from `.github/workflows/release.yml`.
- The current release asset names are:
  - `numi-v<version>-x86_64-unknown-linux-gnu.tar.gz`
  - `numi-v<version>-x86_64-apple-darwin.tar.gz`
  - `numi-v<version>-aarch64-apple-darwin.tar.gz`
  - `numi-v<version>-x86_64-pc-windows-msvc.zip`
- The current Homebrew tap support for `numi` installs from source tarballs and uses `cargo install`.
- Homebrew only needs macOS and Linux support for `numi`.
- The tap update script needs deterministic asset names and checksums.

## Constraints

- The Homebrew formula must not depend on Rust for `numi`.
- The formula must install from GitHub release assets, not from the source archive.
- The formula must support:
  - macOS Apple Silicon
  - macOS Intel
  - Linux x86_64
- Windows release assets remain published on GitHub Releases but are not consumed by Homebrew.
- The solution should stay simpler than full bottle generation.

## Options Considered

### 1. Recommended: binary formula from GitHub release archives

Use the existing `numi-v<version>-<target>.tar.gz` archives directly in `Formula/numi.rb`, with per-platform `url` and `sha256`.

Why this wins:

- matches the current release workflow output
- removes the `rust` build dependency from Homebrew installs
- keeps the tap automation straightforward
- avoids bottle-specific structure and metadata work

### 2. Full bottle workflow

Convert release packaging to true Homebrew bottles and declare `bottle do` blocks.

Why this is rejected for now:

- requires a different packaging format and more Homebrew-specific release logic
- adds complexity that is unnecessary for the current goal

## Design

### Release asset contract

`numi` will keep the following archive names stable:

- `numi-v<version>-x86_64-unknown-linux-gnu.tar.gz`
- `numi-v<version>-x86_64-apple-darwin.tar.gz`
- `numi-v<version>-aarch64-apple-darwin.tar.gz`

Each archive contains a top-level directory named `numi-v<version>-<target>/` and a `numi` binary inside it.

This naming convention becomes an external contract for the Homebrew tap update script.

### Homebrew formula shape

`Formula/numi.rb` becomes a binary formula with:

- explicit `version`
- conditional `url` and `sha256` values by platform
- no `rust` build dependency
- an `install` method that extracts the chosen archive and installs the `numi` binary

The formula should:

- use the macOS Apple Silicon archive on Apple Silicon Macs
- use the macOS Intel archive on Intel Macs
- use the Linux x86_64 archive on Linux

### Tap update script

`scripts/update-formula.sh` will gain binary-formula rendering for `numi`.

For `numi`, the script will:

1. accept a release tag like `v0.1.0`
2. derive the three supported release asset URLs from the tag
3. download each archive
4. compute each archive SHA-256
5. render `Formula/numi.rb` with explicit version, URLs, and checksums

The script should continue to support existing formulas without changing their behavior.

### Version tracking

Because the formula will no longer point to a source tarball URL, the tap script must not infer the current version by parsing `archive/refs/tags/...`.

For `numi`, the script should read the explicit `version` field from the current formula when deciding whether to preserve the previous formula as `numi@<version>.rb`.

## Files Expected To Change

In `oops-rs/numi`:

- `.github/workflows/release.yml`
- `docs/crates-io-release.md`

In `oops-rs/homebrew-tap`:

- `Formula/numi.rb`
- `scripts/update-formula.sh`
- `README.md`

## Verification

Before completion:

- run the tap update script for `numi` against a real tag
- confirm the generated formula points to the GitHub release asset URLs
- confirm the generated formula contains three checksums for the supported platforms
- confirm no existing tap formulas are changed unintentionally
- verify the `numi` release workflow still uploads assets with the expected names

## Risks

- If the release asset names change, Homebrew updates will break.
- If a release omits one of the supported archives, tap automation will fail.
- If the formula’s platform conditionals do not match Homebrew’s execution model, installs could fetch the wrong archive.

## Mitigations

- Treat the asset names as a documented release contract.
- Keep the release workflow naming logic explicit and stable.
- Render the formula from downloaded assets rather than handwritten checksums.
